# DatasetReader 0.7.0

## Step 4: Unified .raw Parsing — MBI Fields and DL Detection Orchestration (PR-3)

Reference: `step04_raw_unified_parsing.md`

### Added

#### `parsers/raw_file_parser.py`

- **`HeaderField.IMG_SEGMENT`** (bytes 8–12, `np.int32`): New enum member for
  parsing the image segment identifier from the .raw header.

- **`HeaderField.JOB_ID`** (bytes 28–32, `np.int32`): New enum member for
  parsing the job identifier from the .raw header. Required by MBI position
  adjustment in the orchestrator.

- **`DiePosX` and `DiePosY` extraction in `_build_metadata()`**: Die position
  coordinates are now read directly from the .raw header (bytes 96–103) and
  populated in `RawImageMetadata.DiePosX`/`DiePosY`. Previously these fields
  remained `None` after parsing; now they reflect the header-encoded values.

- **`ImgSegment` and `JobId` extraction in `_build_metadata()`**: Both new
  header fields are extracted and included in the returned `RawImageMetadata`.

- **`scanDir=3` alignment with arceus in `_normalize_scan_dir()`**: Replaced
  the previous `cv2.ROTATE_180` transform for `ScanDirection.WEST` with a
  vertical flip (`img[::-1, :]`), matching arceus (`image_io.py`). All four
  scan-direction transforms now use vectorised numpy slicing with no `cv2`
  dependency in this path.

#### `orchestrators/dl_detection_orchestrator.py`

Step 4 `.raw` orchestration functions replacing the placeholder TODO block:

- **`is_mbi_mode(image_path)`** — Detects MBI vs SBI acquisition mode from the
  filename. MBI filenames have 7+ dash-separated segments; SBI have 6 or fewer.
  Replaces `HMIImageCoordProcessor.is_cs_mode()` from arceus.

- **`adjust_position_for_mbi(metadata)`** — Computes MBI-corrected image
  position using `JobId`, `ImgWidth`, and `PixelSizeX`. Returns original
  positions unchanged for SBI images (where `JobId` is `None`). Replaces
  `HMIImageCoordProcessor._adjust_image_positions_for_mbi()`.

- **`get_die_position(image_x, image_y, metadata, img_pos_x, img_pos_y, use_offsets)`**
  — Converts pixel coordinates to die-relative coordinates (µm) using parsed
  header/footer fields and optionally applying global/local footer offsets.
  Replaces `HMIImageCoordProcessor.getDiePos()`.

- **`get_gds_position(image_x, image_y, metadata, img_pos_x, img_pos_y, y_flip, use_offsets)`**
  — Converts pixel coordinates to GDS coordinates using image position and
  pixel sizes. Replaces `HMIImageCoordProcessor.getGdsPos()`.

- **`get_wafer_position(image_x, image_y, metadata, img_pos_x, img_pos_y, y_flip, use_offsets)`**
  — Converts pixel coordinates to wafer coordinates (µm). Replaces
  `HMIImageCoordProcessor.getWaferPos()`.

- **`create_image_mapping_dict(raw_images_folder, image_ext)`** — Globs a
  directory of `.raw` files and builds a `{wildcard_key: image_stem}` mapping
  where the 4th dash-segment of each filename is replaced with `'*'`. Replaces
  `dzf_io.py:create_image_mapping_dict()`.

- **`load_raw_image(raw_image_path, mix_weights)`** — Convenience wrapper
  around `RawFileParser.parse()`. Returns `(image_2d_uint8, RawImageMetadata)`.

- **`load_ref_images(raw_image_path, ref_image_dict, mix_weights)`** — Loads
  the two reference `.raw` images associated with a defect image using a
  caller-supplied `{image_stem: [ref1_stem, ref2_stem]}` mapping. Replaces
  `HMIImageCoordProcessor.read_ref_images()`.

#### `orchestrators/__init__.py`

New Step 4 exports added to the public API: `is_mbi_mode`,
`adjust_position_for_mbi`, `get_die_position`, `get_gds_position`,
`get_wafer_position`, `create_image_mapping_dict`, `load_raw_image`,
`load_ref_images`.

### Changed

- **`_build_metadata()` in `raw_file_parser.py`**: Extended to extract and
  populate `DiePosX`, `DiePosY`, `ImgSegment`, and `JobId` from the header.
  Callers receive richer `RawImageMetadata` objects with no API change
  required.

### Notes

**Coordinate transform design:** `get_die_position()`, `get_gds_position()`,
and `get_wafer_position()` are stateless functions that accept
`(image_x, image_y, metadata, img_pos_x, img_pos_y)` explicitly. In arceus
these were instance methods accessing `self.*` attributes with mutated state.
The stateless design is more testable and aligns with DatasetReader's
functional style.

**`DiePosX`/`DiePosY` enrichment coordination:** These fields are now populated
by the parser directly from the .raw header. If the DatasetManager enrichment
pipeline also sets `DiePosX`/`DiePosY`, verify that it does not unconditionally
overwrite the parser-populated values.

**`scanDir=3` resolved:** `ScanDirection.WEST` now uses a vertical flip
(`img[::-1, :]`), matching the arceus reference behaviour. The previous
`cv2.ROTATE_180` transform has been removed.

**Patch alignment stub:** `crop_defect_patch()` and `align_and_crop()` are
deferred — they require `hmpybind.imageprc` (C++ binding unavailable in the
mlplatform environment). Placeholder comments are in the orchestrator.

### Migration Guide

```python
from asml.hmi.DatasetReader.orchestrators import (
    is_mbi_mode,
    adjust_position_for_mbi,
    get_die_position,
    get_wafer_position,
    create_image_mapping_dict,
    load_raw_image,
    load_ref_images,
)
from pathlib import Path

raw_path = Path("data/image-12345-1-0012-42-0-1.raw")

# 1. Detect MBI mode
mbi = is_mbi_mode(raw_path)

# 2. Load image and metadata
image, metadata = load_raw_image(raw_path, mix_weights=(0.5, 0.3, 0.2, 0.0))

# 3. Adjust position for MBI
img_pos_x, img_pos_y = (metadata.ImgPosX, metadata.ImgPosY)
if mbi:
    img_pos_x, img_pos_y = adjust_position_for_mbi(metadata)

# 4. Convert defect pixel coordinates to wafer coordinates
wafer_x, wafer_y = get_wafer_position(
    image_x=320, image_y=256,
    metadata=metadata,
    img_pos_x=img_pos_x,
    img_pos_y=img_pos_y,
)

# 5. Build image mapping dict from folder
mapping = create_image_mapping_dict(Path("data/raw_images/"))
```
